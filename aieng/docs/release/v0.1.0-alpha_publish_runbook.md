# v0.1.0-alpha owner publish runbook (#178)

Owner-triggered, reviewable runbook for publishing the **v0.1.0-alpha** release:
TestPyPI dry-run → PyPI → GHCR image → git tag → GitHub release notes, plus
rollback notes and post-publish verification.

> **Honesty boundary — read first.**
> - **Nothing here publishes automatically.** Every `twine upload`, `docker push`,
>   `git tag`, and `gh release` step is an explicit owner action. CI builds and
>   *smoke-tests* the artifacts; it does not push them.
> - This is an **alpha**: advisory, evidence-recording, **not engineering-certified**
>   and under no stability/semver guarantee. Release notes must keep that wording
>   (see [release_blocker_audit.md](release_blocker_audit.md) and the
>   `*_CLAIM_BOUNDARY` constants in `aieng-ui/backend/app/honesty.py`).

Packages and current versions (both alpha-pinned):

| Package | Source | Version | Build backend |
|---|---|---|---|
| `aieng-format` | `aieng/` | `0.1.0a2` | hatchling |
| `aieng-workbench-mcp` | `aieng-ui/backend/` | `0.1.0a2` | (see its `pyproject.toml`) |

`aieng-workbench-mcp` depends on `aieng-format`, so **publish `aieng-format`
first** on each registry.


Expected artifacts in `aieng/dist/`:
- `aieng_format-0.1.0a2-py3-none-any.whl`
- `aieng_format-0.1.0a2.tar.gz`

**Resource-bundling check (the #144 concern — schemas must ship).** The wheel
packages `src/aieng/schemas/*.json` (per `[tool.hatch.build.targets.wheel]
artifacts`). Verify the schemas are present:

```bash
python - <<'PY'
import glob, zipfile
whl = glob.glob("dist/aieng_format-0.1.0a2-py3-none-any.whl")[0]
names = zipfile.ZipFile(whl).namelist()
schemas = [n for n in names if n.endswith(".schema.json")]
print("schema files in wheel:", len(schemas))           # expected: 44
print("claim_map shipped:", any("claim_map" in n for n in names))  # expected: False
PY
```

Expected: **44 schema files**, and `claim_map shipped: False`. The latter is
**intentional** — `claim_map.schema.json` lives only in the top-level legacy
`aieng/schemas/`, not in the packaged `src/aieng/schemas/`, so the installed
package cannot load it. This is consistent with the alpha posture that claim-map
/ claim-advancement is out of scope (see [release_blocker_audit.md](release_blocker_audit.md)).

## 2. TestPyPI dry run (owner-triggered)

Validate the upload + install path on TestPyPI before touching real PyPI.

```bash
# aieng-format first (dependency order)
cd aieng
python -m twine upload --repository testpypi dist/*

cd ../aieng-ui/backend
python -m twine upload --repository testpypi dist/*
```

Then install from TestPyPI into a **clean** venv and smoke it (real PyPI is the
fallback for the `aieng-format` dependency, which may not be on TestPyPI):

```bash
python -m venv /tmp/aieng-testpypi && source /tmp/aieng-testpypi/bin/activate
pip install --index-url https://test.pypi.org/simple/ \
            --extra-index-url https://pypi.org/simple/ \
            aieng-workbench-mcp==0.1.0a2
python -c "import aieng; print('aieng-format import OK')"
# headless MCP smoke (tool listing) — see MCP_SETUP.md for the exact entry point
```

**Expected:** clean install, imports succeed, MCP server lists the canonical
tools (`aieng.agent_readme`, `aieng.list_projects`, `cad.execute_build123d`,
`cae.prepare_solver_run`).

## 4. GHCR image publish

**Automated (#177).** The `docker-smoke` workflow publishes the all-in-one image
to GHCR **from `main`, after the full build + smoke passes** — never from a PR.
It pushes `ghcr.io/armpro24-blip/aieng-workbench:latest` plus an immutable
`sha-<commit>` tag, authenticating with the built-in `GITHUB_TOKEN` (no extra
secret). So the owner triggers a publish simply by **merging to `main`** with the
Docker paths changed (or via the workflow's manual `workflow_dispatch` on `main`);
a published image is always a smoke-validated image.

**Manual fallback** (e.g. to push an explicit version tag like `0.1.0-alpha`):

```bash
IMAGE=ghcr.io/armpro24-blip/aieng-workbench
docker build -t "$IMAGE:0.1.0-alpha" .
echo "$GHCR_PAT" | docker login ghcr.io -u <owner> --password-stdin
docker push "$IMAGE:0.1.0-alpha"
```

Verify by pulling the published tag on a clean host and re-running the Docker
smoke checks (MCP `tools/list` + managed-approval fail-safe — #175 — and the
`/data` restart-persistence check — #176).

## 5. Git tag + GitHub release notes (owner-triggered)

```bash
git tag -a v0.1.0-alpha -m "v0.1.0-alpha"
git push origin v0.1.0-alpha
gh release create v0.1.0-alpha --title "v0.1.0-alpha" --notes-file <notes.md> --prerelease
```

Release notes **must**:
- Mark the release **`--prerelease`** / alpha.
- Quote the honesty boundary: `.aieng` records evidence and context; it does
  **not** certify engineering correctness and does **not** advance engineering
  claims automatically. No stability/semver guarantee at alpha.
- List the published artifacts (PyPI versions + GHCR tag) and the
  "connect your agent" snippets from `MCP_SETUP.md` pointed at the published
  install paths.

## 6. Post-publish verification

Reuse the packaged external-agent dogfood path (#143 / #179):

- Install from the **published** PyPI artifacts (or pull the GHCR image) in a
  clean environment — not the maintainer source checkout.
- Connect at least one external MCP agent; confirm `aieng.agent_readme`,
  prompts/resources, and the tool list are visible.
- Exercise an approval-gated CAD mutation, viewer refresh, a face-pointer
  handoff, and a CAE preflight that reports missing inputs honestly.
- Record the evidence packet + punch-list on #143 / #179.

> Do **not** claim solver execution unless `cae.run_solver` actually ran and
> result artifacts exist. This is adoption evidence, not certification.

---

## 7. Rollback / remediation

- **PyPI / TestPyPI:** a version is immutable and **cannot be overwritten**. If a
  bad artifact is uploaded, **yank** it (`pip`-invisible but installable by exact
  pin) via the project page or `twine`/web UI, and publish a fixed
  **`0.1.0a3`** — never reuse `0.1.0a2`. Test on TestPyPI first to avoid burning
  a real version.
- **GHCR:** delete or re-tag the bad image version in the GHCR package UI / API;
  consumers pinning the old digest are unaffected until they re-pull.
- **Git tag / release:** `gh release delete v0.1.0-alpha` and
  `git push origin :refs/tags/v0.1.0-alpha` to remove a premature tag, then
  re-tag the corrected commit.
- If §6 surfaces a blocking gap, **stop**: yank/withdraw what was published, file
  the punch-list, and do not advertise the install paths until a fixed alpha is
  republished.
